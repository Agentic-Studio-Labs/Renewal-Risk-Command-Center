# Demo Runbook

Use this runbook to present the Renewal Risk Command Center demo.

## Demo Goal

Show how Guild can coordinate a renewal-risk workflow across business tools:

- Google Sheets supplies account health data.
- Slack provides the human-in-the-loop approval surface.
- Google Docs stores the approved action plan.
- Guild runs and publishes the agent workflow.

## Before The Demo

Confirm:

- The agent is published and installed in the target Guild workspace.
- Slack, Google Spreadsheets OAuth, and Google Docs OAuth credentials are connected.
- The Slack channel exists and the Guild app can post there.
- The Google Sheet has an `Accounts` tab with data matching `examples/accounts.csv`.
- The current demo spreadsheet ID in `agent.ts` matches the Sheet you want to use.

Optional CLI smoke test:

```bash
guild chat \
  --agent <owner>~<agent-name> \
  --workspace <owner/workspace-name> \
  --once \
  "Run a credentials smoke test only: read Accounts!A:L from spreadsheet <spreadsheet-id> using Google Sheets, then stop. Do not post to Slack or create Google Docs."
```

## Talk Track

### 1. Set Up The Business Problem

"Renewal risk usually lives across multiple systems. CSMs may track customer health in spreadsheets, approvals happen in Slack, and action plans get written up after the fact. This demo shows how a Guild agent can coordinate that workflow while keeping a human approval step in the middle."

### 2. Explain The Current Data Source

"For this demo, Google Sheets stands in for a production data layer. In a production implementation, these fields would come from CRM, finance, product usage, and support systems."

Mention current gaps clearly:

- CRM is not integrated yet.
- Finance and billing systems are not integrated yet.
- Product usage and support telemetry are not integrated yet.
- The spreadsheet is the demo source of truth.

### 3. Trigger The Scan

In Slack:

```text
@Guild run a renewal-risk scan
```

Expected behavior:

- The agent acknowledges the request.
- The agent reads the spreadsheet.
- The agent posts a proposal for the highest-risk account.

### 4. Review The Proposal

Point out that the proposal includes:

- Account name
- Owner
- Renewal date
- ARR
- Health score
- Risk rationale
- Recommended next step
- Approval instructions

This is the human checkpoint. The agent does not claim approval or update a source system before a human approves.

### 5. Approve The Plan

In Slack:

```text
@Guild APPROVE
```

Expected behavior:

- The agent acknowledges approval.
- The agent creates a Google Doc.
- The agent inserts the approved action plan into the document.
- The agent posts the document link back to Slack.

### 6. Close With The Production Path

"The same pattern can be expanded from a spreadsheet demo into a production command center. CRM can provide opportunity and account context, finance can provide ARR and payment risk, product analytics can provide usage signals, and support systems can provide escalation risk. Guild coordinates the workflow and approval flow across those tools."

## Expected Demo Output

For the sample data, the agent should identify:

```text
Nimbus Health
Health Score: 48
Risk Status: At Risk
```

The exact action plan text may vary because it is generated by the model, but it should stay grounded in the spreadsheet data.

## Recovery Notes

If Slack approval does not trigger the agent, include the Guild app mention:

```text
@Guild APPROVE
```

If the Google Doc is created but no link appears, inspect the Guild session events:

```bash
guild session get <session-id>
guild session tasks <session-id>
guild session events <session-id>
```

If credentials fail, reconnect the specific Google integration that failed. Google Drive credentials do not replace Google Sheets or Google Docs credentials.
